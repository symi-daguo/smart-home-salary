# 员工类型路由 Skill（employee-skill-router）

## 目的

在一个公司统一部署 OpenClaw 的场景，员工可能用模糊语音/文字发送指令，例如：

- “帮我报一下单子，万科，五万，九五折”
- “今天装了 10 个开关，在万科项目”
- “我三月份工资多少”

本 Skill 的职责是：

1. 结合 `taxonomy-smart-home` 的词典，把输入归类为：销售上报 / 技术上报 / 工资查询 / 管理引导
2. 将任务路由到最匹配的岗位 Skill（例如 `member-sales`、`member-technician`、`member-salary`、`admin-guide`）
3. 如果缺少关键字段，提出“最少追问”的单轮确认问题

输出规范：不使用 emoji。

## 路由规则（高优先级）

1. 若包含“工资/薪资/发工资/本月工资”等关键词：
   - 路由到：`member-salary`
2. 若包含“安装/调试/售后/维修/上门/故障”等关键词：
   - 路由到：`member-technician`
3. 若包含“销售/收款/订单/报单/提成/折扣/回款”等关键词：
   - 路由到：`member-sales`
4. 若包含"整体/全员/导出/审批/结算/异常/统计/预警"等管理倾向关键词：
   - 路由到：`admin-guide`
5. 若包含"测量/工勘/勘测/信息记录/窗帘下单/窗帘/轨道/房间数量"等管理录入关键词：
   - 路由到：`admin-guide`（测量工勘与窗帘下单为管理端功能，需 OWNER/ADMIN 在 Web 后台操作）
5. 无法判定时：
   - 先问一个二选一问题（最省 token）：
     - “你要做的是：1) 上报销售 2) 上报安装/调试 3) 查询工资。回复序号即可。”

## 最小确认策略

路由到具体 Skill 后，如果缺字段：

- 销售上报缺“金额”：追问“金额多少（元）？”
- 技术上报缺“数量”：追问“数量多少（个/套）？”
- 技术上报缺“类型”：追问“这是安装、调试还是售后？回复其中一个词。”
- 若项目名称不确定：
  - 要求下游 Skill 使用 `GET /api/projects` 做候选列表（<=3 个时只让用户回序号）
- 若产品名称不确定：
  - 要求下游 Skill 使用 `GET /api/products` 做候选列表（<=3 个时只让用户回序号）

## 与本项目的联动边界

本 Skill 不直接调用后端业务接口，不直接创建数据；只负责分类与最小确认，并把任务交给以下 Skills：

- MEMBER：`member-sales`、`member-technician`、`member-salary`
- ADMIN/OWNER：`admin-guide`

## 统一交互约定：提交前必须二次确认（强制）

本路由 Skill 需要确保下游所有“会写入数据库”的 MEMBER 类 Skills（例如 `member-sales`、`member-technician`）遵守同一条交互约定：

- 在真正调用 `POST /api/*/my` 写入前，必须先把解析后的“表单摘要”汇总给员工查看
- 员工必须回复关键字 **“确认”** 才允许提交
- 员工回复“取消/不对/重填”时不得提交

该约定的目的：

- 避免语音识别/口语歧义造成误填
- 保障员工对“写入服务器、进入系统报表/工资/预警”的后果有明确知情与确认
